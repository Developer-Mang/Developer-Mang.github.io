브라우저의 랜더링 과정

브라우저에 페이지가 나오기 까지

어떤 페이지에 접속하기 위해서는 브라우저를 실행하고 일련의 과정을 거쳐야지 접속할 수 있다.

그 일련의 과정이란

  1. 리소스 요청
    해당 페이지에 접근하고 브라우저에 출력하기 위한 리소스를 서버에 요청해야한다. 브라우저에서 제공하는 URL 입력창에 URL을 입력하면 해당 URL를 DNS를 통해 IP주소로 변환하여 해당 주소를 갖는 서버에 리소스를 요청한다. URL만 입력시, 자동적으로 해당 주소의 index.html파일을 요청한다.

  2. HTML 파싱과 DOM 생성
    서버가 응답하여 HTML 파일 데이터를 전달 받으면 해당 데이터를 해석하는 과정인 파싱을 렌더링 엔진이 하기 시작한다.
    파싱은 우선, 받은 데이터를 HTML 파일의 charset을 기준으로 문자로 변환된다. > 변환된 문자를 문법적 의미를 갖는 토큰으로 분해한다. > 토큰을 객체로 변환하여 노드를 생성한다. > 이 노드들의 관계를 반영하여 트리 자료 구조인 DOM을 생성한다.

    1. CSS 파싱과 CSSOM 생성
      HTML 파싱 중에 link 태그나 style 어트리뷰트를 만나면 HTML 파싱과정을 중단하고 해당 CSS 파일을 서버에 요청하며 HTML 파싱과정과 동일한 CSS 파싱 과정을 거친다. 파싱과정을 거치면 CSSOM을 생성한다.

    2. 자바스크립트 파싱과 실행
      CSS 파싱과 동일하게 script 태그를 만나면 해당 HTML 파싱을 중단한다. 그리고 src 속성이 있을 경우 해당 주소에 있는 리소스를 요청하여 자바스크립트 엔진에게 제어권을 넘겨 받아 해당 스크립트 파일과 script 태그 안에 코드들을 HTML/CSS 파싱과 동일하게 파싱과정을 거쳐 AST를 생성한다. 그리고 AST를 기반으로 자바스크립트의 기능을 수행하는 인터프리터가 해석할 수 있는 바이트 코드를 생성하여 인터프리터에게 전달하고 인터프리터는 그 바이트 코드를 실행한다.
      이때, DOM이나 CSSOM을 변경할 수 있다.

  3. 렌더 트리 생성
    그렇게 완성된 DOM이나 CSSOM를 결합하여 렌더링할 수 있는 렌더 트리를 생성한다. 이때 브라우저 화면에 표시 되지 않는 노드들은 포함하지 않는다.

  4. 페인팅 처리
    렌더 트리를 사용하여 HTML 요소의 레이아웃을 계산하여 화면 픽셀에 렌더링하여 출력한다. 이를 패인팅 처리라고 한다.
    페인팅 처리는 다음과 같은 상황에서 반복하여 실행될 수 있다.

    • 자바스크립트에 의한 노트 추가 및 삭제
    • 자바스크립트에 의한 HTML 요소의 스타일 변경
    • 브라우저 윈도우에 의한 리사이징

    자바스크립트에 의해 DOM이나 CSSOM이 변경되어 리페인팅이 되는 경우를 리플로우 또는 리페인트라고 한다.

HTTP/1.1과 HTTP/2의 차이

HTTP/1.1은 커넥션당 1개의 요청과 응답을 처리한다. HTTP/2는 커넥션당 여러개의 요청과 응답을 처리하는 응답 다중화를 지원한다.
따라서 HTTP/2가 페이지 로드 속도가 빠르다.

Script 태그의 async/defer 어트리뷰트

HTML 파싱 중 script 태그를 만나면 파싱을 멈추고 해당 자바스크립트 파싱 및 실행을 진행한다. 이때 외부에서 파일을 로드하는 경우, async와 defer 속성을 사용하면 파일 로드를 비동기적(HTML 파싱과 동시에)으로 진행할 수 있다.

두 속성 다 비동기적으로 파일을 로드하지만 실행 시점에 차이가 있다. async는 자바스크립트 파싱과 실행을 자바스크립트 로드가 끝나자 마자 실행하지만 defer 자바스크립트 파싱과 실행을 HTML 파싱이 끝난 후에 실행한다.
